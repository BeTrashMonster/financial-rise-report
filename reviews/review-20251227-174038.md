#### HIGH-004: Unbounded Database Query in ReportController ğŸŸ 
**Anti-Pattern:** API-002 (Missing Pagination)
**Severity:** HIGH
**Impact:** Memory exhaustion, slow queries, potential DoS

**Location:**
- `financial-rise-backend/src/controllers/reportController.ts:49`

**Issue:**
```typescript
// Line 49-51
const responses = await AssessmentResponse.findAll({
  where: { assessmentId },
});
```

This query loads ALL responses for an assessment without any limit. While assessments may have a bounded number of questions (~50), this is still an unbounded query pattern.

**Also Found In:**
- `financial-rise-backend/src/controllers/reportController.ts:271`
- `financial-rise-backend/src/services/validationService.ts:161`
- `financial-rise-backend/src/services/progressService.ts:18`

**Recommended Fix:**
Since assessments have a known bounded number of responses, this is lower risk than general list endpoints. However, best practice is to add a reasonable limit:

```typescript
const responses = await AssessmentResponse.findAll({
  where: { assessmentId },
  limit: 500,  // Safety limit - assessments shouldn't exceed this
  order: [['createdAt', 'ASC']]
});

if (responses.length >= 500) {
  logger.warn(`Assessment ${assessmentId} has suspiciously high response count`);
}
```

#### MED-004: CSP Unsafe-Inline in Production ğŸŸ¡
**Anti-Pattern:** SEC-003 (Missing Security Hardening)
**Severity:** MEDIUM
**Impact:** Reduced XSS protection

**Location:**
- `financial-rise-backend/src/middleware/security.ts:19`

**Issue:**
```typescript
// Line 19
"'unsafe-inline'", // TODO: Replace with nonce-based approach in production
```

Content Security Policy allows inline scripts/styles, weakening XSS protection.

**Recommended Fix:**
1. Implement nonce-based CSP:
```typescript
app.use((req, res, next) => {
  res.locals.nonce = crypto.randomBytes(16).toString('base64');
  next();
});

helmet.contentSecurityPolicy({
  directives: {
    scriptSrc: ["'self'", (req, res) => `'nonce-${res.locals.nonce}'`],
    styleSrc: ["'self'", (req, res) => `'nonce-${res.locals.nonce}'`],
  }
});
```

2. Update templates to use nonce:
```html
<script nonce="${nonce}">...</script>
<style nonce="${nonce}">...</style>
```

## New Anti-Patterns Discovered

### NEW-001: Mock Data TODOs in Production Controllers ğŸŸ¡ MEDIUM
**Description:** Controllers using hardcoded mock data with TODO comments indicating incomplete integration with actual services.

**Impact:** Production code doesn't deliver full functionality, reports show fake data.

- Hardcoded mock objects (discProfile, phaseResults, etc.)

- Use feature flags for incomplete features
- Fail fast with clear errors rather than serving mock data

**Priority:** MEDIUM
**Discovered:** 2025-12-27 (This review)

### NEW-002: Empty Test File Stubs ğŸŸ  HIGH
**Description:** Test files that exist in the project structure but contain only skeleton code with TODO comments, giving false impression of test coverage.

**Impact:** False confidence in test coverage, regression risk, delays finding bugs.

**Check For:**
- Test files with only `describe()` and `it()` shells
- Test bodies containing only `// TODO: Implement actual tests`
- High ratio of test files to actual implemented tests

**Correct Approach:**
- Don't commit empty test stubs unless actively working on them
- Use test coverage tools to identify untested code
- Follow TDD: write tests before implementation
- Mark incomplete test suites clearly in documentation

**Priority:** HIGH
**Discovered:** 2025-12-27 (This review)

---

### NEW-003: Dependency Version Mismatch (Zod) ğŸŸ  HIGH
**Description:** Backend and frontend using different major versions of the same validation library, or specifying non-existent package versions.

**Impact:** Type incompatibilities, runtime errors, developer confusion, build failures.

**Check For:**
- Same dependency with different major versions in frontend/backend package.json
- Package versions that don't exist in npm registry
- Type definition mismatches between projects

**Correct Approach:**
- Use workspace/monorepo tools to enforce consistent versions
- Maintain shared package.json for common dependencies
- Verify package versions exist before committing
- Use exact versions (no ^ or ~) for critical shared dependencies

**Priority:** HIGH
**Discovered:** 2025-12-27 (This review)

### NEW-004: Console Logging in Production Code ğŸŸ¡ MEDIUM
**Description:** Using console.log/error/warn instead of structured logging library for production code.

**Impact:** Unstructured logs, hard to parse, no log levels, no metadata, poor production debugging.

**Check For:**
- `console.log`, `console.error`, `console.warn`, `console.debug` in src/ directories
- Exclude test files and development scripts

**Correct Approach:**
- Use structured logging library (Winston, Pino, Bunyan)
- Include context/metadata in logs
- Use appropriate log levels (info, warn, error, debug)
